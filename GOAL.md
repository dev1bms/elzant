# GOAL.md — نظام الدعوات عبر واتساب + تحسين بصري لـ elzant.com

> **الجمهور:** هذا المستند مكتوب ليُسلَّم مباشرة إلى **Claude Code** كأمر عمل (Work Order) كامل.
> **المشروع:** `elzant.com` — موقع زفاف محمود ورينان (Django 6 · SQLite · Tailwind v4 · عربي RTL بالكامل).
> **دورة العمل:** إضافة نظام إرسال دعوات عبر **WhatsApp Cloud API** بلوحة عائلية بسيطة + رفع الجانب البصري.
> **الحالة:** النسخة العامة (الموقع + جدار التهاني) تعمل بالفعل؛ هذه الدورة تبني طبقة الدعوات فوقها دون كسر ما يعمل.

## 0) التعاقد التنفيذي (اقرأه أولاً — إلزامي)

اتبع تسلسلاً متدرّجاً ولا تقفز إلى التنفيذ:

```
Discovery → Planning → Implementation → Review → Commit
```

- **لا تنفّذ أي أمر مدمّر** (حذف بيانات/جداول/migrations/ملفات) إطلاقاً دون شرحٍ واضح وانتظار تأكيد صريح مني.
- **راجع وافهم قبل أن تكتب.** أنجز المرحلة 1 (المراجعة) بالكامل واعرض ملخّصها قبل أي تعديل كود.
- بعد كل مرحلة، **توقّف واعرض ملخّصاً قصيراً** للتغييرات (ملفات، migrations، مخاطر) قبل الانتقال.
- الكود والمعرّفات بالإنجليزية؛ **كل نص ظاهر للمستخدم بالعربية RTL**.
- **توافق Django 6:** انتبه لأي فروق عن Django 4/5 (خصوصاً `path` converters، `check --deploy`، إعدادات الأمان). لا تستخدم واجهات مهجورة.
- حافظ على **فلسفة الخصوصية** الحالية: لا تحليلات، لا تتبّع طرف ثالث في الواجهة العامة. طبقة الدعوات وحدها تتكامل مع Meta (خادم-لخادم)، ولا يُحقَن أي سكربت تتبّع في صفحات الزوّار.
- **بيئة العمل:** فعّل البيئة الافتراضية، شغّل `python manage.py check` و`check --deploy` كبوّابات سلامة بعد كل مرحلة. لا يوجد Linter مُهيّأ.

## 2) لقطة الوضع الحالي (مبني مسبقاً — لا تُعِد بناءه)

راجعتُ الكود؛ هذه اللبنات موجودة **الآن** ويجب البناء فوقها لا تكرارها:

- **`WeddingConfig`** (singleton, `pk=1`, `.get()`): يحوي الأسماء، الموعد، القاعة، و**قوالب الرسائل** (`default_whatsapp_message_template`, `default_email_subject`, `default_email_body_template`) — قابلة للتعديل من الأدمن.
- **`WeddingGuest`**: فيه بالفعل `invitation_token` (عبر `secrets.token_urlsafe(16)`)، `phone_number`, `email`, `group`, `guest_count`, و**دورة حالة** `Status`: `draft → ready → sent_whatsapp → sent_email → opened → greeted`، بالإضافة إلى `last_opened_at` و`invited_at`.
- **`Greeting`** + **`GreetingSuggestion`**: جدار تهاني بمراجعة لاحقة (post-moderation).
- **الأدمن** (`core/admin.py`): يولّد بالفعل **رابط `wa.me` جاهز لكل مدعو** عبر `_whatsapp_message()` + `quote()`، ومعاينة بريد، وإجراءات `mark_sent_whatsapp` / `export_links_csv` / …
- **تتبّع الفتح جاهز:** فيو `invitation(request, token)` في `core/views.py` يضبط `last_opened_at` والحالة `opened` عند فتح `/i/<token>/`، ويخزّن التوكن في الجلسة ليُربط بأي تهنئة تُكتب.
- **`render_message(template, mapping)`** في `core/utils.py`: يستبدل `{{ key }}` — المتغيّرات المتاحة: `guest_name`, `invitation_link`, `groom_name`, `bride_name`, `wedding_date`, `venue_name`.
- **النشر:** Gunicorn خلف Cloudflare Tunnel (بروكسي موثوق) — لدينا بالفعل مسار عام آمن مناسب لاستقبال **webhook** من Meta.

**الخلاصة:** ننتقل من «رابط wa.me يدوي في الأدمن» إلى «إرسال آلي عبر Cloud API من لوحة عائلية، مع تتبّع تسليم/قراءة حقيقي ونسبة الدعوة لصاحبها».

## 4) الهدف العام لهذه الدورة (Goal)

بناء **طبقة دعوات** تمكّن أهل العريس وأهل العروس من:
1. تسجيل الدخول إلى **لوحة بسيطة على الجوال**، وإضافة مدعو (اسم + هاتف) وإرسال دعوته **آلياً عبر واتساب** بضغطة.
2. أن تحمل الرسالة **رابطاً خاصاً لكل مدعو** (`/i/<token>/`)، مع ظهور **حالة التسليم والقراءة وفتح الرابط** في اللوحة.
3. أن يرى كل مُرسِل **مدعويّه وحالاتهم**، مع خيار عرض **كل المدعوين ومن دعا كلاً منهم**.
4. عند إدخال رقم **مدعوٍّ مسبقاً**: تظهر رسالة «مدعو بواسطة فلان»، ولا يُرسَل تكرار إلا بعد **تأكيد صريح**.

بالتوازي: **رفع الجانب البصري** للوحة والصفحات العامة، وتقديم **خطة أفكار مبتكرة** قابلة للتنفيذ لاحقاً.

## 5) نموذج البيانات (تغييرات المخطّط)

> اكتب migrations نظيفة ومتدرّجة. أضف الحقول بقيم افتراضية آمنة. **لا تحذف** أي حقل قائم في هذه الدورة.

**(أ) توسعة `WeddingGuest`:**
- `phone_e164 = CharField(max_length=20, blank=True, db_index=True)` — الرقم مُطبَّعاً (E.164 بلا `+`)، أساس كشف التكرار.
- `invited_by = ForeignKey(settings.AUTH_USER_MODEL, null=True, blank=True, on_delete=SET_NULL, related_name="invited_guests")`.
- `wa_message_id = CharField(blank=True)` — معرّف آخر رسالة قالب من Meta.
- `wa_status = CharField(choices=[none, queued, sent, delivered, read, failed], default="none", db_index=True)`.
- `delivered_at`, `read_at` = `DateTimeField(null=True, blank=True)`.
- `send_count = PositiveSmallIntegerField(default=0)` · `last_sent_at = DateTimeField(null=True, blank=True)`.

> **ملاحظة:** أبقِ `invitation_status` القائمة كما هي (تدفّق العرض)، واعتبر `wa_status` **إشارة مكمّلة** خاصة بالتسليم لا بديلاً. حدّث الاثنين بتناسق.

**(ب) نموذج جديد `InviterProfile` (OneToOne مع User):**
- `user = OneToOneField(User)` · `display_name = CharField` (الاسم الظاهر بالعربية) · `side = CharField(choices=[groom, bride, both])` · `can_view_all = BooleanField(default=False)` (صلاحية رؤية كل المدعوين) · `phone = CharField(blank=True)`.

**(ج) نموذج جديد `WhatsAppConfig` (singleton مثل `WeddingConfig`):**
- `enabled = BooleanField(default=False)` (مفتاح تشغيل/إيقاف الإرسال الحيّ).
- `phone_number_id = CharField` · `waba_id = CharField` (WhatsApp Business Account ID).
- `default_template_name = CharField` · `template_lang = CharField(default="ar")`.
- `default_country_code = CharField(default="20")` (مصر؛ لتطبيع الأرقام المحلية).
- `test_recipient = CharField(blank=True)` (رقم اختبار الأدمن).
- **الأسرار لا تُخزَّن هنا:** `WHATSAPP_TOKEN`, `WHATSAPP_APP_SECRET`, `WHATSAPP_VERIFY_TOKEN` تُقرأ من `.env` فقط.

**(د) نموذج جديد `WhatsAppTemplate`** (لإدارة قوالب عربية متعددة من الأدمن):
- `name` (اسم القالب في Meta) · `language` · `category` (utility/marketing) · `body_preview_ar` (نص للعرض) · `variables_map` (JSON: ترتيب المتغيّرات) · `is_approved = BooleanField` · `active`.

**(هـ) نموذج جديد `MessageLog`** (سجل تدقيق لكل إرسال):
- `guest = FK(WeddingGuest)` · `sender = FK(User, null=True)` · `template_name` · `wa_message_id` · `status` · `error = TextField(blank=True)` · `payload = JSONField(blank=True, null=True)` (اختياري) · `created_at`.

## 6) تكامل WhatsApp Cloud API

**الهدف:** وحدة خادم-لخادم لإرسال قوالب واتساب واستقبال تحديثات الحالة.

### 6.1 وحدة العميل `core/whatsapp.py`
- `send_template(to_e164, template_name, lang, components)` → POST إلى Graph API (`/{phone_number_id}/messages`) بالتوكن من البيئة؛ يعيد `wa_message_id` أو يرفع خطأً واضحاً.
- `normalize_phone(raw, default_cc)` → E.164 بلا `+` (يعالج الأصفار البادئة، الفراغات، `00`، والرمز الدولي). أعِد `None` للأرقام غير الصالحة.
- `verify_webhook_signature(request)` → تحقّق من `X-Hub-Signature-256` عبر `WHATSAPP_APP_SECRET` (HMAC-SHA256). ارفض غير الموقّع.
- تعامل مع كل استدعاء شبكي بمهلة (timeout) ومعالجة أخطاء صريحة؛ سجّل الفشل في `MessageLog`.

### 6.2 نقطة الـ Webhook `/webhooks/whatsapp/`
- `GET`: تحقّق الاشتراك (`hub.mode`, `hub.verify_token`==`WHATSAPP_VERIFY_TOKEN`, `hub.challenge`).
- `POST`: `csrf_exempt` + تحقّق التوقيع، ثم عالِج أحداث `statuses[]` (`sent/delivered/read/failed`) وحدّث `WeddingGuest` (`wa_status`, `delivered_at`, `read_at`) و`MessageLog` عبر مطابقة `wa_message_id`. عالِج أيضاً `messages[]` الواردة (اختياري: رد آلي — §9).
- أعِد `200` سريعاً دائماً (Meta تعيد المحاولة عند غير 200).

### 6.3 قالب الدعوة العربي (Utility) — للاعتماد في Meta
> **مهم:** الرسائل التي يبدؤها العمل تتطلّب **قالباً معتمداً مسبقاً**. صنِّفه **Utility** (أرخص، وغالباً مجاني داخل نافذة 24 ساعة) لا Marketing حيث أمكن.

**البنية المقترحة (متغيّران + زر رابط ديناميكي):**
- **Body:**
  ```
  السلام عليكم {{1}} 🌿
  يسعدنا دعوتكم لحضور حفل زفاف محمود ورينان.
  تفضّلوا بفتح دعوتكم الخاصة من الزر أدناه — بانتظار تشريفكم.
  ```
  `{{1}}` = اسم المدعو.
- **زر (URL ديناميكي):** الأساس `https://elzant.com/i/` والمتغيّر `{{1}}` = التوكن → الرابط النهائي `https://elzant.com/i/<token>/`.
  > اجعل مسار الدعوة يتسامح مع/بدون الشرطة المائلة الأخيرة (تحقّق من `APPEND_SLASH` أو أضف نمطاً بديلاً) حتى لا يكسر زرُّ واتساب الرابطَ.
- وفّر **3 صياغات عربية بديلة** (رسمي / دافئ عائلي / مختصر) في `WhatsAppTemplate` ليختار منها الأدمن (§9 يقدّم النصوص).

### 6.4 الإعداد والاختبار
- أضف مفاتيح `.env`: `WHATSAPP_TOKEN`, `WHATSAPP_APP_SECRET`, `WHATSAPP_VERIFY_TOKEN`، وحدّث `.env.example` و`deploy/.env.production.example` (بلا قيم حقيقية).
- **زر «إرسال رسالة اختبار»** في أدمن `WhatsAppConfig`: يرسل القالب إلى `test_recipient` ويعرض النتيجة (نجاح/`wa_message_id`/الخطأ). استخدم رقم الاختبار الذي توفّره Meta + رقمك قبل التوثيق الكامل.
- ابدأ و`enabled=False` (وضع آمن): في هذا الوضع لا يُرسَل شيء فعلياً بل يُسجَّل «سيُرسَل» — لاختبار التدفّق دون تكلفة.

## 7) اللوحة العائلية البسيطة (قلب هذه الدورة)

**الهدف:** واجهة **جوال-أولاً**، عربية، على هوية الموقع، يستخدمها غير التقنيين بسهولة.

### 7.1 المسارات (تطبيق/بلوبرنت جديد `panel`)
| المسار | الغرض | الوصول |
|---|---|---|
| `/panel/login/` · `/panel/logout/` | دخول/خروج المُرسِل | عام/مصادَق |
| `/panel/` | لوحة موجزة: عدّاداتي + آخر الدعوات + زر «دعوة جديدة» | مُرسِل |
| `/panel/guests/` | قائمة المدعوين (`?scope=mine` افتراضياً، `?scope=all` إن سُمح) | مُرسِل |
| `/panel/guests/add/` | إضافة (اسم+هاتف) → كشف تكرار → إرسال | مُرسِل |
| `/panel/guests/<id>/` | تفاصيل مدعو + خطّ زمني للحالة | مالكه أو `can_view_all` |
| `/panel/guests/<id>/resend/` | إعادة إرسال (بتأكيد) | مالكه أو `can_view_all` |

### 7.2 الصلاحيات والعزل
- كل المسارات `login_required` + التحقّق من امتلاك `InviterProfile` (أو عضوية مجموعة `inviters`).
- **عزل البيانات:** فلترة `WeddingGuest.objects.filter(invited_by=request.user)` افتراضياً؛ لا يُظهر `scope=all` إلا لمن `can_view_all=True`. امنع وصول مُرسِل لتفاصيل مدعو ليس له عبر الـ URL (تحقّق في الفيو، لا في القالب فقط).
- في عرض «الكل»: أضف عمود **«دعاه: {display_name}»** ووسم **الجهة** (عريس/عروس).

### 7.3 تدفّق الإضافة + كشف التكرار (المتطلّب الحرج)
1. المُرسِل يُدخل الاسم + الهاتف → `normalize_phone`. أظهر خطأً عربياً واضحاً للأرقام غير الصالحة.
2. ابحث عن `WeddingGuest` بـ `phone_e164`.
3. **إن وُجد:** لا تُرسِل مباشرة. أظهر بطاقة تنبيه:
   > «هذا الرقم مدعوٌّ مسبقاً بواسطة **{display_name}** بتاريخ {invited_at}، الحالة: {…}. لإرسال دعوة مكرّرة اضغط **تأكيد**.»
   مع خيارين: **«إعادة الإرسال للمدعو نفسه»** (المُوصى به — نفس الرابط) أو **«إضافة كسجل مكرّر»**. لا إجراء بلا `confirm=1` صريح.
4. **إن لم يوجد (أو بعد التأكيد):** أنشئ/حدّث المدعو (`invited_by=request.user`)، أرسل القالب عبر `core/whatsapp.py`، أنشئ `MessageLog`، اضبط `invitation_status=sent_whatsapp` + `invited_at` + `last_sent_at` + `send_count += 1`.
5. أعِد للمُرسِل حالة فورية (نجاح/فشل) وزرّ «دعوة جديدة».

### 7.4 لوحة الحالة (تتبّع «قرأ الدعوة ولا لأ»)
اعرض لكل مدعو **خطّاً زمنياً/شارات** مشتقّة من إشارتين:
- **من واتساب (webhook):** أُرسلت → سُلّمت → **قُرئت** (`read_at`).
- **من موقعنا:** **فتح الرابط** (`last_opened_at`) → **كتب تهنئة** (`greeted`).
لخّص في اللوحة عدّادات: (مُرسَلة · مُسلّمة · مقروءة · فُتح الرابط · هنّأ) — «دعواتي» والكل.

## 8) الجانب البصري (منطقة تركيز رئيسية)

> ابنِ فوق نظام التصميم القائم في `tailwind/input.css` (لوحة «Sunset on the Mediterranean / درب القاهرة»، خطوط Aref Ruqaa/Cairo/Tajawal، الأزرار `.btn-primary` النبيذية، السطوح `.panel`). **لا تُدخِل CDN؛ استخدم Tailwind build القائم.**

**لوحة العائلة (جديدة — اجعلها راقية لا «أدمن خام»):**
- شاشة دخول أنيقة بهوية الزفاف (خلفية دافئة، اسمان بخط العرض، حقول `.field`).
- **لوحة تقدّم بصرية:** حلقات/أشرطة تقدّم لقُمع الحالات (مُرسَل → مُسلّم → مقروء → فُتح → هنّأ) بألوان النظام.
- **بطاقة مدعو** بخطّ زمني أفقي بشارات (✓ رمادي/ذهبي/نبيذي) بدل جدول جاف.
- **معاينة حيّة للدعوة** قبل الإرسال (اسم المدعو داخل نص القالب + الرابط).
- أزرار كبيرة ملائمة للإبهام، حركة واحدة هادئة (اتّبع `prefers-reduced-motion` القائم).

**الصفحات العامة (تلميع خفيف، لا إعادة هيكلة):**
- صفحة `/i/<token>/`: لمسة ترحيب باسم المدعو أرقى (already personalized) + دعوة واضحة لكتابة تهنئة.
- توليد **QR** لكل دعوة داخل اللوحة (لمشاركتها ورقياً/شخصياً) — مكتبة `qrcode` المتوفّرة في `node_modules` أو سيرفر-سايد.
- تحسين اتّساق الظل/الحواف الذهبية عبر اللوحة لتطابق البطاقة القابلة للمشاركة.

**مبدأ:** التايبوغرافي والمساحة البيضاء يقودان الإحساس الفاخر، لا الزخرفة — تماشياً مع `docs/design-review/`.

## 9) أفكار مبتكرة (Backlog مُرتّب — خارج نطاق التنفيذ الآن، للتخطيط)

**قريبة العائد / منخفضة الكلفة:**
1. **RSVP / تأكيد الحضور:** زر «سأحضر / أعتذر» في `/i/<token>/` + عدّاد رؤوس في اللوحة (كان خارج نطاق MVP — أعلى قيمة مضافة الآن).
2. **تذكيرات مجدولة:** قالب Utility تذكيري قبل 3 أيام ويوم الحفل لمن لم يفتح/يؤكّد (Cron/management command).
3. **إضافة للتقويم (.ics)** + رابط خرائط مباشر في صفحة الدعوة.
4. **ردّ آلي على الرسائل الواردة** (webhook messages): من يرد على الدعوة يصله رابط التفاصيل/الموقع تلقائياً (داخل نافذة 24 ساعة → مجاني).

**متوسطة:**
5. **تحليلات لطيفة للعائلة:** نسبة الفتح/القراءة لكل مُرسِل، «لوحة شرف» ودّية (من أعلى استجابة).
6. **استيراد جماعي CSV** للأرقام مع تطبيع وكشف تكرار دفعة واحدة.
7. **إدارة الطاولات/الجلوس** ورمز حضور (check-in QR) على الباب.
8. **جدار تهاني حيّ** يُعرض على شاشة أثناء الحفل (وضع كامل الشاشة).

**لاحقة:**
9. **تعدّد اللغات فعلياً** (الإنجليزية) — البنية جاهزة (LocaleMiddleware/locale).
10. **دعوة فيديو/صوت** شخصية قصيرة لكل مدعو.

## 10) الاختبار (بوّابة إلزامية قبل أي Commit إغلاق)

- **وحدات (fast):** `normalize_phone` (أرقام مصرية/فلسطينية/أردنية + حالات حدّية)، كشف التكرار، `render_message`، تحقّق توقيع الـ webhook، **عزل الصلاحيات** (مُرسِل لا يرى مدعوي غيره).
- **تكامل:** تدفّق الإضافة→الإرسال مع **محاكاة (mock)** لـ Cloud API؛ معالجة `statuses` من الـ webhook تُحدّث الحالة صحيحاً.
- **يدوي:** `enabled=False` أولاً (تدفّق كامل بلا تكلفة) → ثم إرسال اختبار حقيقي لرقمك → افتح الرابط وتأكّد من انتقال الحالة إلى «فُتح» و«قُرئ».
- **سلامة:** `python manage.py check` و`check --deploy` نظيفان؛ لا أسرار في Git؛ `collectstatic` يعمل بعد أي تغيير CSS.
- `core/tests.py` حالياً هيكل فارغ — اكتب أول suite حقيقي لهذه الميزة.

## 11) الأمان والخصوصية (حرِج — لا تساوم)

- **الأسرار في `.env` فقط** (التوكن/App Secret/Verify Token). لا تُخزَّن في قاعدة البيانات ولا تُطبَع في اللوجات ولا تُرفَع لـ Git.
- **الـ webhook:** تحقّق التوقيع إلزامي؛ ارفض غير الموقّع؛ لا تثق بالحمولة قبل التحقّق.
- **عزل المُرسِلين:** افرض الفلترة في طبقة الفيو (queryset)، لا في القالب فقط.
- **الهاتف بيانات حسّاسة:** لا يُعرَض علناً أبداً؛ يظهر داخل اللوحة للمصرّح لهم فقط. أضِف إمكانية حذف مدعو + بياناته.
- **حدّث إشعار الخصوصية** (`WeddingConfig.privacy_notice_short` و`templates/core/privacy.html`): صار النظام يخزّن الهاتف ويُرسِل عبر خدمة Meta — يجب ذكر ذلك بوضوح.
- **الموافقة (Opt-in):** سياسة واتساب تتطلّب أن يكون المستقبِل موافقاً على التلقّي. أرسل لمدعوّين معروفين للعائلة فقط، وابدأ تدريجياً، وراقب **جودة الرقم (Quality Rating)** لتفادي الحظر.
- تحديد المعدّل (rate limiting) يبقى مسؤولية الحافة/Cloudflare (كما هو موثّق) — لا تُضِف throttling داخل العملية.

## 12) المخاطر والتحذيرات

1. **اعتماد القالب يستغرق وقتاً** (قد ساعات–أيام) وقد يُرفَض؛ جهّز صياغات بديلة وابدأ الاعتماد مبكراً.
2. **الرقم المُرسِل رقم أعمال** لا رقم العائلة الشخصي — بعض المدعوين قد لا يعرفونه؛ اضبط **اسم العرض للنشاط** باسم الزفاف/العائلة، ووثّق النشاط، واكتب نصاً واضحاً.
3. **التكلفة:** الفوترة **لكل رسالة** منذ 1 يوليو 2025، وتُحدَّد بمقدّمة رقم المُستقبِل (سوق مصر خُفِّض مؤخراً). قوالب **Utility** أرخص بكثير من Marketing، و**مجانية** غالباً داخل نافذة الخدمة 24 ساعة — لذا صنّف القالب Utility وأبقِ الحجم معقولاً.
4. **الـ webhook يحتاج HTTPS عاماً** — متوفّر عبر Cloudflare Tunnel؛ تأكّد من وصول Meta إليه ومن `CSRF_TRUSTED_ORIGINS`.
5. **زر الرابط الديناميكي** حسّاس للشرطة المائلة الأخيرة — اختبر الرابط النهائي فعلياً.
6. **تطبيع الأرقام** مصدر أخطاء شائع (أصفار بادئة/رمز دولي مزدوج) — غطِّه باختبارات.
7. **نافذة 24 ساعة:** الرسائل الحرّة (بلا قالب) لا تجوز إلا داخلها؛ خارجها لا بد من قالب معتمد.
8. **توافق Django 6:** تحقّق من أي تغييرات في إعدادات الأمان/الـ converters قبل الاعتماد على سلوك قديم.

> **ملاحظة تحقّق:** آليات Cloud API (نقاط النهاية، حقول الـ webhook، فئات القوالب) قد تتغيّر — **تحقّق من توثيق Meta الرسمي وقت التنفيذ** ولا تعتمد على الذاكرة.

## 13) ترتيب البناء المقترح + Git

**المراحل (كل مرحلة تُغلَق ببناء نظيف + اختبارات خضراء + عرض ملخّص):**
1. **مراجعة وفهم** → `docs/REVIEW.md` (لا كود). — *`docs: full code review & architecture map`*
2. **المخطّط والهجرات:** حقول `WeddingGuest` + `InviterProfile` + `WhatsAppConfig` + `WhatsAppTemplate` + `MessageLog` + أدمنها. — *`feat(model): invitations schema (inviter, wa config/templates, message log)`*
3. **وحدة Cloud API + webhook** (خلف `enabled=False`) + اختبار الأدمن. — *`feat(wa): cloud api client + verified status webhook`*
4. **اللوحة العائلية:** دخول + إضافة + كشف تكرار + إرسال + قوائم «دعواتي/الكل». — *`feat(panel): family invite panel with dedupe + attribution`*
5. **تتبّع الحالة والعرض** (خطوط زمنية + عدّادات). — *`feat(panel): delivery/read/opened tracking`*
6. **التلميع البصري** للّوحة والصفحات العامة + QR. — *`feat(ui): panel visual system + qr`*
7. **الخصوصية والنصوص** + `check --deploy` + `.env.example`. — *`chore(sec): privacy copy, env, deploy checks`*
8. **مراجعة نهائية** + وسم. — *`Release: WhatsApp invitations v1`*

**اصطلاحات:** فروع `feature/<اسم>`، رسائل `feat(scope): …` / `fix(scope): …` / `chore(scope): …`. لا تدمج فرعاً إلا ببناء نظيف واختبارات خضراء. اعرض الـ diff قبل أي merge.

```bash
git checkout -b feature/wa-invitations
# … بعد كل مرحلة مستقرّة (تُبنى وتمرّ):
git add -A && git commit -m "feat(panel): family invite panel with dedupe + attribution"
# إغلاق الدورة:
git checkout main && git merge --no-ff feature/wa-invitations
git tag -a v1.2.0-invites -m "WhatsApp Cloud API invitations + family panel"
```

## 14) النتيجة المتوقعة عند الاكتمال

- ✅ أهل العريس/العروس يدخلون لوحة جوال بسيطة، يضيفون مدعواً (اسم+هاتف) ويرسلون دعوته **آلياً عبر واتساب** بضغطة.
- ✅ الرسالة تحوي **رابطاً خاصاً لكل مدعو**، وتظهر في اللوحة حالة **التسليم/القراءة/فتح الرابط/كتابة التهنئة**.
- ✅ **تعدّد مُرسِلين**: كلٌّ يرى دعواته وحالاتها، وخيار «كل المدعوين + من دعاهم» للمصرّح لهم.
- ✅ عند تكرار رقم مدعوٍّ مسبقاً: تحذير **«مدعو بواسطة فلان»** ولا إرسال إلا بـ**تأكيد**.
- ✅ إعدادات واتساب التشغيلية من الأدمن، والأسرار في `.env`، وزر **اختبار إرسال** يعمل.
- ✅ لوحة وصفحات **أرقى بصرياً** على هوية الموقع، وخطة أفكار مبتكرة موثّقة للمراحل التالية.
- ✅ اختبارات خضراء، `check --deploy` نظيف، ولا أسرار في المستودع.
